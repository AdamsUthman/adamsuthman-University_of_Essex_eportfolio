TEAM ACTIVITY: DISCUSSION

Question: What factors determine whether a programming language is secure or not?

Ans: The three main programming language vulnerabilities, according to Cifuentes & Bierman (2019), are buffer errors, injection mistakes, and information leak flaws. No programming language is immune to all three of these errors, as they showed in their research, hence there is no such thing as a "secure language."

Question: Could Python be classed as a secure language? Justify your answer.

Ans: Although having some more elegant memory management features than certain other languages,  Python is nonetheless prone to several security flaws, such as overflow issues and injection attacks (especially in the case of web frameworks). As a result, Python cannot be regarded as a secure language.

Question: Python would be a better language to create operating systems than C. Discuss.

Ans: Python is a fantastic language that can be used for a number of reasons since it is a high-level language, making it simpler to code. It also has substantial library support. It is slower, though, because it is an interpreted language. Contrarily, C is a compiled language that gives greater flexibility in addition to being faster. It would be more appropriate to design an operating system based on how it handles memory. C is therefore a superior language to use for developing an operating system.

PORTFOLIO COMPONENT: Programming language concepts

1. What is ReDOS and what part do ‘Evil Regex’ play?

The Regular Expression Denial of Service (ReDoS) attack takes use of the fact that most Regular Expression implementations occasionally encounter extreme circumstances that make them operate slowly (exponentially related to input size). When a program using a Regular Expression (Regex) encounters these unusual circumstances, an attacker can make the application hang for a very lengthy time.

2. What are the common problems associated with the use of regex? How can these be mitigated?

One of the problems associated with using regrex is the incorreect use of wildcards Larson & Kirk (2018). When utilizing wildcards directly, individuals frequently take the short cuts without carefully considering what undesirable characters are included. By employing a variety of regex checker tools and educating programmers about potential regex mistakes and their effects, such errors can be reduced.

3. How and why could regex be used as part of a security solution?

Regex is an extremely effective technique for validating user input, including passwords and other sensitive data. For instance, using username in the password can easily be blocked using regex. Therefore, the programmer must take great care to prevent weaknesses in the regex in order to prevent potential assaults made with malicious regex.

Skills/Knowledge Gained

  • Regular expressions (regex) are a sequence of characters used to check if a character pattern exists in a text or not.
  • Regular Expression Denial of Service (ReDoS) is a type of Denial of Service attack in which the attacker inputs a regex that causes the system to hang for a long time.
  • Common regex mistakes (which could result in an evil regex attack) include: an excessive dependence on wildcard (using wildcard rather than considering other options), bad range, bad anchor usage, delimiter mismatches, wrong repeats, and improper character usage.

PORTFOLIO COMPONENT: Exploring the Cyclomatic Complexity’s Relevance Today

The Cyclomatic Complexity is commonly considered in modules on testing the validity of code design today. However, in your opinion, should it be? Does it remain relevant today? Specific to the focus of this module, is it relevant in our quest to develop secure software? Justify all opinions which support your argument and share your responses with your team.

Cyclomatic Complexity (CC) is still important today, in my opinion, at least in some parts. The complexity of a source code should not, however, be determined exclusively by this metric; rather, it should be used in conjunction with other metrics that consider the complexity of the individual branches and the complexity of the conditions themselves. The mentioned factors may have a significant impact on the program's runtime because the quantity of decision points is only one component of complexity. Nonetheless, in my opinion, CC still provides a general picture of the level of complexity, and a high CC may be a sign that the code needs to be refactored.
